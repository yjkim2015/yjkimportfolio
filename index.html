  • Redis 캐시 조회 지연 시 에러 페이지로 전환되는 불안정성
  • 빅스마일데이 트래픽 대비 안정적인 TPS 확보 필요


어려웠던 점과 극복 방안
  • 아키텍처 통합: Spring Boot 3.4 기반 멀티 모듈 구조(Aggregator, Core, Biz, Extension, Admin, Data)로 통합 API 설계
  • SP 종속성 제거: DB 전용 API 서비스 구축, SP 호출 로직을 별도 모듈로 분리하여 비즈니스 로직과 데이터 조회 로직의 결합도 감소 → 원인 분석과 성능 모니터링 용이
  • Redis Pub/Sub 구현 → Admin 설정 변경 시 캐시 자동 갱신
  • Warm-up 구현: Spring ApplicationListener와 k8s StartupProbe를 활용해 JIT 초기화 지연 해결
  • CQRF 패턴 적용해 조회 모델과 저장 모델을 분리 → 조회 모델은 Redis에서 조회하여 성능 최적화 → 저장 모델은 Oracle과 MongoDB에 기록하여 영속성 보장
  • 이중 캐시 전략 (Global Cache + Local Cache) 도입
  • 성능 테스트(K6) 점진적 부하 증가(Ramp-up) → Stable TPS / Limit TPS 측정 → 단일 Pod 및 다중 Pod 환경에서 트래픽 시뮬레이션 → 목표 TPS: 빅스마일데이 대비 최대 2000TPS


성과
  • 통합된 API 모델을 설계하여 데이터 일관성 유지, 변경 사항이 생겨도 앱의 배포를 최소화 하며, 다양한 클라이언트에서 동일한 로직을 재사용할 수 있는 기반을 마련
  • SP 호출 서비스 분리로 유지보수성 및 문제 추적 용이
  • Local + Global 캐시 계층 설계로 성능 안정화 및 장애 내성 확보
  • CSRF 패턴 기반 모델 분리로 성능 최적화
  • 배포 시 Warm-up 자동화로 초기 Latency 제거
  • 성능 테스트 결과 ： 다중 Pod 환경 확장성 확인 → 목표 TPS 2000 대응 가능
